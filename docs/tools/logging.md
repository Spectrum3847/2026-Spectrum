# Logging and Data Analysis

## What are Logs?

Logs are records of events, data, and system behavior generated by the robot code during operation. They are essential for understanding how the robot is performing, diagnosing issues, and optimizing its behavior.

## Reading Logs

Logs are most useful when searching for errors in the robot's operation.

### Common Usage

*   **Brownouts**: Checking logs for power-related issues that cause the robot to unexpectedly reset or lose functionality.
*   **Failure Analysis**: Identifying the sequence of events leading up to a system failure.

### Verbal Representations (Log Levels)

Logs typically use different levels to indicate the severity or type of information:

*   **Red (Errors)**:
    *   Indicates critical problems that prevent normal operation.
    *   **Caution**: Read carefully, as sometimes these can be expected messages during robot boot-up (e.g., when the robot is looking for a network table host to connect to).
    *   Other times, red errors might indicate motors stopped working/connecting, or a battery brownout during processing.

*   **Yellow (Warnings)**:
    *   Indicates potential problems or unusual conditions that might not be critical but should be investigated.

## DogLog

DogLog is a logging library that we extend through our `Telemetry` class.

### Telemetry (Our Custom Logging Class)

`Telemetry` is our abstract class that extends DogLog, providing customized methods for logging.

*   **General Usage**: Used for recording data from robot initializations, hard-coded checks, and robot failures (both at start-up and during runtime).
*   **Values Stored**:
    *   Commands that are run and ended (recorded in the DriverStation Log).
    *   Variables storing different values when needed.
*   **Usage Examples**:
    *   Simply call `Telemetry.log(...)` in a `periodic()` method.
    *   Can also be used in states: `log(...[command])`.
*   **Log Commands**: Record when commands start and end.
*   **Print Commands**: Output information directly to the console or DriverStation.
*   **Logging Alerts**: Provide immediate feedback on significant events.
*   **Using FRC Team 501's logging library**: Our `Telemetry` class builds upon their established logging framework.

### Documentation

Integrating `Telemetry` for logging into your code is straightforward and crucial for debugging and performance analysis.

*   **Calling `Telemetry.log()`**:
    *   **In `periodic()` methods**: For values that change frequently and need continuous monitoring (e.g., sensor readings, motor outputs).
        ```java
        // Example in a subsystem's periodic() method
        @Override
        public void periodic() {
            Telemetry.log("Swerve/RobotPoseX", getRobotPose().getX());
            Telemetry.log("Swerve/RobotPoseY", getRobotPose().getY());
        }
        ```
    *   **In Commands/States**: To log events, command starts/ends, or specific state transitions.
        ```java
        // Example in a command's initialize() or end() method
        public class MyCommand extends CommandBase {
            @Override
            public void initialize() {
                Telemetry.log("Command/MyCommand", "Started");
            }
            @Override
            public void end(boolean interrupted) {
                Telemetry.log("Command/MyCommand", "Ended. Interrupted: " + interrupted);
            }
        }
        ```
    *   **For critical events**: Such as initialization, configuration changes, or error conditions.
        ```java
        // Example in Robot.java constructor
        Telemetry.print("--- Robot Init Starting ---");
        // ...
        Telemetry.print("--- Robot Init Complete ---");
        ```
*   **Log Categories**: Use clear, hierarchical names for your log entries (e.g., `"Subsystem/ValueName"`) to organize data in logging tools like `AdvantageScope`.
*   **`Telemetry.print()`**: Use `Telemetry.print()` for messages intended for the DriverStation console, especially for significant events like mode transitions.
*   **`PrintPriority`**: For `Telemetry.print()`, you can specify a `PrintPriority` (e.g., `NORMAL`, `HIGH`, `CRITICAL`) to filter messages if needed.
